# Architecture`n`nThe `c-abstract-http` library is designed to offer a clean, unified API over disparate native networking implementations. `n`n## Core Philosophy`n`n1. **Native By Default**: The library avoids shipping thick protocol implementations (like libcurl) when the underlying OS already provides a robust HTTP client stack (like WinHTTP/WinINet on Windows or CFNetwork on Apple).`n2. **C89 Purity**: All code is written in strict C89 to maximize portability. C++ features and syntax are strictly avoided.`n3. **Robust Error Propagation (The `int` Return Paradigm)**: `n   - All functions that perform an operation, allocate memory, or query state **must return an `int`** representing an exit code (e.g., `0` for success, `EINVAL`, `ENOMEM`, etc.).`n   - If a function needs to return a value (e.g., retrieving a header), it does so by accepting an `out` pointer (e.g., `const char** out_val`).`n   - This allows the caller to perfectly trace failure paths without ambiguous `NULL` returns.`n4. **Zero-Bloat Windows Headers**:`n   - `c-abstract-http` enforces a strict ban on `#include <windows.h>`.`n   - Instead, the codebase utilizes leaner headers like `<windef.h>`, `<winnt.h>`, `<winbase.h>`, `<winnls.h>`, and `<winerror.h>`.`n   - This prevents macro collisions, drastically reduces compile time, and keeps the namespace clean for consuming applications.`n5. **MSVC Safe CRT Features**:`n   - On MSVC environments (`_MSC_VER`), dangerous CRT functions (`sprintf`, `strcpy`, `fopen`) are avoided.`n   - The library employs `#if defined(_MSC_VER) && !defined(__INTEL_COMPILER)` compiler guards to selectively utilize `_s` extensions (`sprintf_s`, `strcpy_s`, `fopen_s`) to prevent buffer overruns without sacrificing compatibility with GCC, Clang, or MinGW.`n`n## Abstraction Layers`n`n### 1. Target Platforms`n`nc-abstract-http targets:`n- **Windows**: MSVC (2005, 2022, 2026), MinGW, Cygwin`n- **macOS / iOS**: Clang / Apple Frameworks`n- **Linux / POSIX**: GCC / Clang`n- **FreeBSD**: Clang`n- **Android**: NDK`n- **WebAssembly**: Emscripten`n`n### 2. Target Network Libraries`n`nThe dispatcher layer evaluates the compiled platform and wires up the backend's specific `send` and `send_multi` implementations. Native default backends are preferred to reduce binary size and use system certificates:`n`n- **WinHTTP**: The modern standard for Windows networking. Uses `WinHttpSendRequest`.`n- **WinINet**: The legacy standard for older Windows environments or specialized proxy/caching setups.`n- **CFNetwork**: Uses `CFNetwork` and `Foundation` types for native execution on macOS and iOS.`n- **libcurl**: The fallback for POSIX systems (Linux, BSD) leveraging `libcurl`.`n- **libuv / libevent**: Modern POSIX backend leveraging `libuv / libevent`, ideal for Node.js environments.`n- **libsoup3**: Modern POSIX backend leveraging `libsoup3`, ideal for GTK4 environments.`n- **libfetch**: FreeBSD and POSIX backend leveraging `libfetch`, ideal for BSD environments.`n- **HttpURLConnection**: Specialized networking integrations for Android JNI environments.`n- **Emscripten Fetch API**: The WebAssembly backend leveraging Emscripten's Fetch API to bridge network calls to the browser's `fetch`.`n`n### 3. Cryptography Integration`n`nTo preserve the `"Native By Default`" philosophy without locking consumers into a heavy OpenSSL dependency, the library dynamically injects cryptography routines natively into the underlying HTTP abstraction via a unified CMake macro (`c_abstract_http_apply_crypto`).`nIf a backend like `libcurl` or `libevent` is used, its configuration is mapped downwards (e.g. bridging `C_ABSTRACT_HTTP_USE_WOLFSSL` -> `CURL_USE_WOLFSSL`). This allows developers to seamlessly pivot between the following crypto libraries without ever modifying their application's code:`n`n- **MbedTLS**`n- **OpenSSL**`n- **LibreSSL**`n- **BoringSSL**`n- **wolfSSL**`n- **s2n-tls**`n- **BearSSL**`n- **Schannel**`n- **GnuTLS**`n- **Botan**`n- **CommonCrypto**`n- **wincrypt**`n`n### 4. CMake Options`n`nCMake configuration drives the abstraction mapping:`n- `C_ABSTRACT_HTTP_HEADER_ONLY`: Use header-only STB-style library`n- `C_ABSTRACT_HTTP_STATIC_CRT`: Use Static CRT (/MT, /MTd)`n- `C_ABSTRACT_HTTP_SHARED_CRT`: Use Shared CRT (/MD, /MDd)`n- `C_ABSTRACT_HTTP_UNICODE`: Use UNICODE`n- `C_ABSTRACT_HTTP_ANSI`: Use ANSI (Multi-Byte)`n- `C_ABSTRACT_HTTP_MULTI_THREADED`: Multi-threaded`n- `C_ABSTRACT_HTTP_SINGLE_THREADED`: Single-threaded`n- `C_ABSTRACT_HTTP_LTO`: Enable Link Time Optimization`n- `C_ABSTRACT_HTTP_USE_LIBSOUP3`: Use libsoup3 instead of libcurl`n- `C_ABSTRACT_HTTP_USE_LIBUV`: Use libuv instead of libcurl`n- `C_ABSTRACT_HTTP_USE_LIBEVENT`: Use libevent instead of libcurl`n- `C_ABSTRACT_HTTP_USE_LIBFETCH`: Use libfetch instead of libcurl`n- Crypto flags: `C_ABSTRACT_HTTP_USE_MBEDTLS`, `C_ABSTRACT_HTTP_USE_OPENSSL`, `C_ABSTRACT_HTTP_USE_LIBRESSL`, `C_ABSTRACT_HTTP_USE_BORINGSSL`, `C_ABSTRACT_HTTP_USE_WOLFSSL`, `C_ABSTRACT_HTTP_USE_S2N`, `C_ABSTRACT_HTTP_USE_BEARSSL`, `C_ABSTRACT_HTTP_USE_SCHANNEL`, `C_ABSTRACT_HTTP_USE_GNUTLS`, `C_ABSTRACT_HTTP_USE_BOTAN`, `C_ABSTRACT_HTTP_USE_COMMONCRYPTO`, `C_ABSTRACT_HTTP_USE_WINCRYPT`.
